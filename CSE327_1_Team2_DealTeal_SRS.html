    1.4 Product Scope

    The "AR Deals" service and specifically its mobile application provides a new, more convenient way of finding the best deals around the user's current position. It eliminates the need of typing any text on-screen whenever the user wants to look for deals, allowing the user to simply point the phone's camera towards their surroundings and view deals available in real-time with the help of placeholders or "deal bubbles" on the points of interest. The application also learns the user's preferences and shows deals according to the user's previous taste to cater a more personalized experience. Refer to the Monetization Strategies and Technology Innovation sub-sections below for further information.

    2. Overall Description

    2.1 Product Perspective

    2.1.1 Overview:

    The software will consist of a mobile application and an accompanying website. The mobile application will use Augmented Reality and marker-less GPS location technology to find deals in real-time on screen, and also allow users to publish or modify their own deals. When a user searches for a particular deal or looks for a certain type of deal frequently, the application will remember the preference and later will also show similar deals even when the user is not looking for those particular types of deals. The website will use the same dataset and allow users to add/edit deals, but will be limited to finding the deals via a location search instead of using AR.

    In order the get location data, the mobile application will use the mobile phone's built-in GPS chip to find the location co-ordinates of the user. This will allow the application to identify nearby deals and notify the user with the data on-screen. The deals will be stored in a database with a key attribute being the automatically generated GPS location co-ordinates along with other supporting information such as titles, ratings, etc. provided by the publisher.

    To seamlessly connect the data, both the platforms will share the same database hosted locally on the devices, and communicate and exchange data whenever needed.

    2.1.2 Monetization Strategy

    There are three direct methods for a prospective monetization scheme, through both the application and the website:

  • Allowing general users to publish their own deals for a monthly subscription fee: A monthly subscription fee will ensure users to post multiple deals to make the most of their investment and in turn, allowing for more traffic through the application. More traffic therefore means a more lucrative platform for bigger business owners to advertise their deals.
  • Putting deals towards the top of the deal allocation pages of both the application and website: Deals can be highlighted and prioritized to be shown towards the top of a location filtered page to ensure more visibility from users. This can be initially achieved or acquired by verified business owners who can highlight their deals in exchange for a monetary value, through transactions outside of the online service.
  • Collaborations with mobile financial services or telecom operators: Such collaborations will make the registration and payment process smoother for the users, and in turn allowing the deals service for a more rapid initial exposure.

    • Other indirect methods may include allowing for advertisements to be displayed towards the bottom of the UI.

    2.1.3 Technology Innovation and Vision

    Augmented Reality is a technology that is still unheard of to many. The AR technology, although may sound gimicky to some, will provide a groundbreaking experience with its ease of use in the near future. Removing the need for the user requiring to input data will "self-disrupt" a lot of tech giants in a positive way. Information can be now provided to potential users or customers without them even looking for it. For example, deals of items that the user might not have been ever interested in can pop up when available, which may intrigue the user to try it out, which would have not been possible in the past.

    Users now seek for a personal, more catered experience in almost every service or product available. With the variety available in the market, a user will feel the most comfortable when he/she is even remotely familiar with the service they will be recieving. Bots are catering to that need, and a machine learning feature that learns the user's preferences tries to provide the user with an experience similar to that very bot. The user will know that the application will only suggest things that the user is interested in, and thus provide a sense of familiarity.

    2.3 User Classes and Characteristics

    Users can either use the mobile application or the website to locate deals. When using the mobile application, the users have the option to use the AR feature to instantly look up deals at a location without the need to search or type any sort of keywords.

    There may be two types of users for this service:

    User seeking deals:

    Such users will use the service when they are actively seeking for a deal on some product they are interested to consume. These users are expected to generally use the AR feature on the mobile application to seek deals when at a location, and as the application will have the real-time AR screen as its Home page, the users are not expected to be tech savvy to be able to use the application to its full potential. This type of user will have limited privileges in the application as they will not be able to publish their own deals until they make the monetary transaction required to do so.

    User publishing deals:

    This type of user will use the service to promote a product or deal they are trying to market. These users will mostly use the application for business purposes and are likely to be interested in trying out newer technological methods to promote their products or services. As this type of user has already completed the monetary transaction (monthly subscription fee) required to be able to publish/edit/delete their own deals, they will have full privileges to update or change information stored in the database.

    Administrators will be able to create or delete user accounts whenever necessary.

    2.4 Operating Environment

    The mobile application will be running on the Android Runtime Environment on an Android based device, commonly on a smartphone. It can be run on Android version 4.1 and above.

    2.5 Design and Implementation Constraints

    As the mobile application's primary feature, the Augmented Reality experience, is entirely dependent on the phone's GPS location accuracy, the accuracy will most likely not be the same for every user. The application will also be dependent on the phone's camera qualities to work smoothly in real time, as a lagging camera will not provide the best experience on the application.

    Also, an internet connection is mandatory for the application as it will convey data from the database to the application almost continuously while the application is open. For this very reason, the fetching of data may see slight delays as the database will also be shared among the application as well as the website.

    2.6 Assumptions and Dependencies

    The following are the assumptions and dependencies that the service takes into account:

    • The GPS location co-ordinates provided by the mobile phone are accurate. It is also assumed to be using the same GPS technology and data pattern on all mobile devices.
    • There may be "ghost deals", meaning locations or deals that may have existed in the past and do not exist anymore, and have not been updated in the subscription renewal period.
    • The AR feature relies on a third-party engine, namely Unity3D, thus becoming a dependency as bugs from their end may hamper user experience.
    • It is also assumed that the mobile device running the application has a capable camera and processing hardware to run the application. If the phone does not have hardware capable enough, the application may not work as it was designed to, or may not work at all.

    3. External Interface Requirements

    3.1 User Interfaces

    3.1.1 Mobile application:

    When a user opens the app for the first time (or after the user logs out of their user account), the user will see a login/signup page, see Figure 3.1. The user will be able to register or log-in by clicking on the designated button.

    When the user has completed signup, a category selection screen will be displayed, see Figure 3.2. The user will be able to select what categories of deals he/she prefers to be offered. But this category window can also be returned back to later whenever required.



    After category selection is completed, the user will be shown a brief loading screen, see Figure 3.3, and will be taken to the AR window with deals displayed in real-time, see Figure 3.4.

    When clicked in the menu button on the top-left corner of the UI, a side bar will pop out listing the following options: Home - to go back to the previous AR window; Categories - to select preferred categories of deals; Your Deals - a list of all deals published by that user; All Deals - a list of all deals available at that time, see Figure 3.5.



    When the user clicks on the Your Deals tab for the first time, the application will notify the user that he/she has not added any deals yet and needs to make a monetary transaction to be able to publish a deal, see Figure 3.6. After the payment is done through an external gateway, the page to add a new deal will appear. It will prompt the user that the deal has been added when the user clicks to save.

    Also from the sidebar, when the user selects the All Deals option, a page listing all deals available, which can be filtered by location, is shown, see Figure 3.7.



    For the next time the user opens the Your Deals tab, all their deals will be listed with the edit and delete icon next to the respective deals, see Figure 3.8, 3.9, 3.10.


    3.2 Hardware Interfaces

    There are no hardware interfaces for the application or the website, as components necessary to run the software come all built-in in the mobile phone or the server running the website.

    3.3 Software Interfaces

    The mobile application uses an open-source graphics engine called "Unity 3D" that enables placeholders to be put on locations in real-time. The phone communicates with the GPS component and fetches location information to determine the user's location, and then with the database in order to get the stored information regarding the deals, which is then shown in the form of placeholders on screen. The application allows the user to read as well as edit the database to add/edit/delete their published deal.

    The website does not have the AR feature, and communicates with the database to display the stored information on the portal. The deals on the website can be filtered by locations, which will also be stored in the database. The website also allows the user to add/edit/delete deals, in turn allowing the user to modify the database.

    3.4 Communications Interfaces

    For a user account to be created, the user requires a Google or Facebook account to register. Data such as user names and date of birth are fetched from the social media accounts and authenticates the user. Only one account can be created with one Facebook/Google account to avoid duplicates. Forms will be maintained manually, with a payment system implemented that will be using gateway services like SSLCommerz to confirm monthly subscriptions of deal publishers.

    5. Other Nonfunctional Requirements

    5.1 Performance Requirements

  • Loading time of the AR feature should be reasonable.
  • Description: The application loading time with the AR screen should not take too much time.
  • Dependency: Unity3D Engine
  • User search should be monitored and learned over time.

  • Dependency: None
