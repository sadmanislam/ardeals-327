1. Introduction

1.1 Purpose

This document will outline in detail the software architecture and design for the DealTeal software. The Software Design Specification helps to convey the particular design and architectural choices that were made for the DealTeal software. It will also showcase different perspectives of the design structure to further understand the system better.

    2. Design Approach

    The software will be developed in the MVC architecture format, i.e., it will consist of 3 tiers - the Presentation or UI tier, the Application or Business tier and the Data or Storage tier. There will be two different interfaces for users - a mobile application and a web counterpart with limited functionality.

    3. Architectural Design

    2.1 Approach

    The type of data we will be dealing with and how it might need to be used in the future should not be constrained or fixed to one format. If limited, then flexibility and scalability of the service of the application will be drastically hampered. To avoid this issue, we opted for the MVC architectural convention as it allows easier modification and maintenance whenever needed. The application server-side code and interface code is kept separate, thus allowing the data to be independently manipulated wherever needed. As methods do not need to be repeated, any change in a particular data will be represented throughout the entire system.

    2.2 Architecture Design Pattern

    2.2 Architecture Design Style

    2.4 Description

    The above architecture diagram represents a hybrid between MVC pattern and the 3-tier style. The system is divided into 3 tiers, with each service connected independently to the server(s). The View and Controller are considered the presentation tier and the Model lives in the Application tier. This allows for more convenient scalability if and when the incoming or outgoing load on the servers and database may increase. There can be multiple servers in the system trying to access the shared database, and changes to the database can be quite frequent. Opting for this hybrid implementation allows for ease of access to the Application tier to services such as REST from other sources.

    Presentation tier:

  • It is the user interface tier that the user can access directly. It allows the user to interact with the software. A key component of our mobile application interface will consist of a 3rd-party engine, namely "Unity3D", and allow the user to experience the Augmented Reality highlight of our software.

    • Application tier:

  • This tier controls the application's functionality throughout. It consists of the servers and logic required to handle data transaction in the system, acting as the bridge between the user's interaction and the data storage.

    • Data tier:

  • The data tier includes the data handling mechanisms, i.e., database servers, backups, etc., as well as the mechanisms to expose the stored data to the tiers above. It does this seamlessly using an Application Programming Interface or API which allows for updates or changes to the data without the Application tier being affected by the changes. Our system uses the built-in database management system in Python, SQLite, instead of a 3rd-party DBMS as the data that will be stored is not very heavy, thus not requiring a server-based solution.

    • 4. Database Design

    4.1 Approach

    The Database required for the DealTeal system is a simple relational database that consists of not many tables or complex relations. We will be using the built-in Python database module called SQLite, as the relationships and queries required to modify and access data in the system is not very complicated.

    4.2 ER Diagram

    4.3 Description

    The database primarily stores data of different types of deals published by different users. There can be three types of users; Administrators - users who are responsible for creating and deleting user accounts, Publishers - users who can publish their own deals, and Consumers - users who opt to browse for deals only.

    A Consumer becomes a publisher when he/she completes the Payment transaction needed to fulfill the requirement. A Publisher can then post a Deal, which will require the Publisher to fill in a form to complete the process. The form includes most of the attributes in the "Deals" table, as well as a detailed Location segment which will also include the Longtitude and Latitude of the deal's location for accuracy.

    So at a glance, a Deal falls into a certain category, and is located at a particular location, and is published by a paid user.

    5. Component Design

    5.1 Cross Platform:

    5.1.1 Class Description

    The DealTeal system does not require subsystems to be implemented for the different platforms, as the server-side data that is needed for the web counterpart of the primary application requires a certain portion of the overall component configuration to work.

    Classes communicate with other classes via controllers. For example, for a User(Publisher) to add a new deal, the AddDeal controller is called to complete the interaction on behalf on the classes. As we are using the MVC architectural pattern, the controller class is basically a class with a few public methods. These methods have a direct 1-to-1 connection with a possible user action, and thus are responsible for processing the necessary data, performing the application code or logic and then showing the corresponding view.

    The Controller classes follow an inheritence hierarchy, where methods from subclasses or child classes can use methods from the superclasses or parent classes. Key distinct actions therefore have their own controllers.
